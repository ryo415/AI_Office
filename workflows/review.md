# Review Workflow

## Purpose

Review an output before the user relies on it or sends it elsewhere.

## When to Use

- The output affects important decisions
- The output will be published or shared
- The output modifies files, systems, or external services
- The user asks for a second opinion
- The output is long or complex

## Roles

- Reviewer
- Chief of Staff
- Engineer, if technical
- Researcher, if factual

## Steps

1. Restate the goal.
2. Check whether the output satisfies the goal.
3. Identify assumptions.
4. Identify missing information.
5. Identify risks.
6. Suggest fixes.
7. Give final recommendation.

## Rules

- Focus on material issues.
- Do not over-review trivial outputs.
- Be explicit about risk level.
- Use `templates/review-report.md` for reusable reviews.

## Output

```markdown
## Review Summary

## Critical Issues

## Minor Issues

## Risks

## Suggested Fixes

## Final Recommendation
```
